\section{Ce que fait le projet}

Tradrly Bot est un \textbf{assistant IA local} contrôlable par voix ou texte,
qui tourne entièrement sur la machine de l'utilisateur (\emph{offline-first}).
Il comprend les commandes en français et en anglais (y compris avec fautes de frappe)
et agit directement sur le PC : ouvrir des applications, régler le volume,
contrôler la luminosité, gérer des fichiers, etc.

\section{Architecture en couches}

\begin{lstlisting}
UI (Electron + React)
       | WebSocket / HTTP
Orchestrateur (Node.js :4000)   <- routeur de commandes
       | HTTP
Couche IA (Python FastAPI :8000) <- NLU / NLP local / STT / TTS
       |
Services externes (Slack, Gmail, Notion...)
\end{lstlisting}

\subsection*{Description des couches}

\begin{itemize}
  \item \textbf{UI} : interface Electron + React, affiche l'avatar animé, l'historique et les réglages. Communique avec l'orchestrateur via WebSocket.
  \item \textbf{Orchestrateur} : cœur Node.js. Reçoit les commandes, applique les règles locales (regex) et délègue à la couche IA si nécessaire.
  \item \textbf{Couche IA} : serveur Python FastAPI. Gère la transcription vocale (STT), la compréhension (NLU + NLP local via spaCy/TextBlob), la génération de réponse, la synthèse vocale (TTS) et la mémoire vectorielle.
  \item \textbf{Services} : intégrations avec des applications tierces (Slack, Teams, Discord, Notion, Gmail, Git, Calendar).
\end{itemize}

\section{Flux d'une commande}

\subsection*{Commande locale (offline)}

\begin{lstlisting}
Utilisateur tape : "ouvre chrome"
  -> Orchestrateur : regex match -> app.open { name: "chrome" }
  -> resolveApp("chrome") -> "chrome" (alias Win)
  -> safeExec('start "" "chrome"') -> Chrome s'ouvre
  -> Reponse : "Ouverture de chrome."
\end{lstlisting}

\subsection*{Commande complexe (NLP local)}

\begin{lstlisting}
Utilisateur tape : "appelle Alice demain c'est urgent"
  -> Orchestrateur -> POST http://localhost:8000/api/analyze
  -> Pipeline NLP : NLU + spaCy + TextBlob
  -> intention + entites (Alice) + priorite (5) + echeance (demain)
  -> reponse generee localement (sans LLM)
  -> UI : badge d'intention + reponse
\end{lstlisting}

\section{Productivité — Gestion d'agenda, tâches et focus}

Une couche productivité complète a été ajoutée à l'interface Productivité.jsx :

\subsection*{Persistance des données (SQLite)}

\begin{itemize}
  \item Table \texttt{app\_state} : stockage clé/valeur générique persistant
  \item Chargement au démarrage : \texttt{GET /api/productivity}
  \item Sauvegarde auto (anti-rebond 400ms) : \texttt{PUT /api/productivity}
  \item Structures persistées : tâches, rappels, agenda, sessions focus, projets
\end{itemize}

\subsection*{Compréhension d'intention (Gemini → OpenAI → Repli local)}

Nouveau module \texttt{ai/app/intent\_action.py} qui transforme le texte libre en action structurée :

\begin{lstlisting}
Entrée : "supprime le rdv de demain"
Sortie : {
  "action": "delete",
  "target": "agenda",
  "content": "",
  "query": "demain",
  "engine": "gemini"
}
\end{lstlisting}

\begin{itemize}
  \item \textbf{Moteur prioritaire} : Gemini (gratuit, clé sur \texttt{aistudio.google.com/apikey})
  \item \textbf{Repli} : OpenAI (si Gemini échoue), puis règles locales par regex
  \item \textbf{Actions détectées} : \texttt{add}, \texttt{delete}, \texttt{complete}, \texttt{list}, \texttt{unknown}
  \item \textbf{Cibles} : \texttt{tasks}, \texttt{reminders}, \texttt{agenda}, \texttt{pomodoro}, \texttt{projects}
  \item \textbf{Tolérance aux fautes} : regex accepte « supprime / efface / arrête / coupe »
\end{itemize}

Endpoint : \texttt{POST /api/productivity/intent} (FastAPI) → relais par l'orchestrateur

\subsection*{Configuration (.env)}

La couche d'intention se paramètre via trois variables d'environnement :

\begin{lstlisting}
GEMINI_API_KEY=...            # cle gratuite (aistudio.google.com/apikey)
GEMINI_MODEL=gemini-2.5-flash # modele valide pour cette cle
INTENT_PROVIDER=auto          # auto | gemini | openai | rules
\end{lstlisting}

\begin{itemize}
  \item \texttt{auto} : essaie Gemini, puis OpenAI, puis le repli local par règles
  \item Sans clé (ou hors-ligne), le repli local par regex assure le fonctionnement
\end{itemize}

\subsection*{Mode Focus (Pomodoro) — Sessions chronométrées}

Interface riche pour les sessions focus / Pomodoro :

\begin{itemize}
  \item \texttt{startPomodoro(text)} : parse la durée (« 50 min », « 1h30 », « demi-heure »…), crée une session
  \item \texttt{parseDuration(raw)} : extraction tolérante de la durée (gère les fautes « mindre », « minutes »…), défaut 25 min
  \item \texttt{recordSession(p, status)} : enregistre en base la session terminée (label, min planifiées, min réalisées, statut)
\end{itemize}

Affichage :
\begin{itemize}
  \item Bandeau de statistiques : Total, Actif, Terminé, X min cumulées
  \item Anneau dégradé cyan→indigo animé (circonférence SVG avec stroke-dashoffset)
  \item Chrono géant \texttt{MM:SS} avec statut (« ● En cours — 50M » / « ⏸ En pause »)
  \item Historique des sessions : liste scrollable (statut, min réalisées/prévues, date) + « Tout effacer »
  \item Boutons : Pause/Reprendre, \textbf{Terminer} (enregistre comme \texttt{done}), Arrêter (enregistre comme \texttt{stopped})
\end{itemize}

Commandes vocales / texte :
\begin{itemize}
  \item « \textbf{démarrer un focus de 50 min} » → lance 50 min auto
  \item « \textbf{démarrer un focus mode} » (sans durée) → 25 min auto
  \item « \textbf{arrête le focus} », « \textbf{stop focus} » → arrête et enregistre
\end{itemize}

\section{En résumé}

\textbf{La compréhension du langage est désormais 100\% locale (NLP).}
Le pipeline spaCy + TextBlob analyse l'intention, les entités, la priorité
et le sentiment sans appel réseau ni clé API. Le contrôle du bureau (volume,
luminosité, applications, fichiers) reste \textbf{100\% offline} via
l'orchestrateur Node.js. GPT-4o n'est plus qu'un \textbf{repli optionnel}.
